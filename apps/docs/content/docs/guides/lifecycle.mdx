---
title: The asset lifecycle
description: Five canonical statuses that every provider's states collapse into.
---

Every provider reports upload and encoding progress differently. Videos SDK collapses
them into **five canonical states** on `Asset.status`:

```ts
type AssetStatus =
  | 'waiting_upload' // asset created, bytes not received yet
  | 'uploading'      // bytes are being received / fetched from a URL
  | 'processing'     // transcoding / packaging
  | 'ready'          // playable
  | 'errored';       // upload or processing failed
```

Reading it is the same on every adapter:

```ts
const { status } = await videos.get(id);
if (status === 'ready') {
  const { hls } = await videos.playback(id);
}
```

## How providers map

| Canonical        | Rehelios                  | Mux         | Cloudflare Stream       | Bunny Stream                     |
| ---------------- | ------------------------- | ----------- | ----------------------- | -------------------------------- |
| `waiting_upload` | `created`                 | —           | `pendingupload`         | `0` Created                      |
| `uploading`      | `uploading`               | —           | `downloading`           | `1` Uploaded                     |
| `processing`     | `queued` · `transcoding`  | `preparing` | `queued` · `inprogress` | `2` Processing · `3` Transcoding |
| `ready`          | `ready`                   | `ready`     | `ready`                 | `4` Finished                     |
| `errored`        | `failed`                  | `errored`   | `error`                 | `5` Error · `6` UploadFailed     |

Every adapter normalizes its provider's own vocabulary into the five canonical states —
Rehelios included: it emits `created` and `failed`, **not** `waiting_upload` or `errored`.
An unknown or new provider status falls back to `processing`, so your app never sees a
value outside the five canonical states.

That fallback is why the left column is the only one you should ever branch on. Compare
`asset.status` against `AssetStatus`; don't hand-write a union of the strings in the
right-hand columns. Guess one wrong and the branch simply never fires — and since the
unknown value normalizes to `processing`, a failed encode reads as "still transcoding"
forever and a `waitForReady` poll never terminates. The provider's raw value is on
`asset.raw` if you genuinely need it.
